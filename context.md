## 1. Общее описание

ВСЗ.CRM — веб-приложение для управления производством моторных лодок ЛОСЬ 400. Объединяет управление клиентской базой, контроль производственных этапов, учёт затрат и штрафов, ИИ-анализ клиентов, хранилище документов и библиотеку инструкций.

Доступ через браузер — с мобильных устройств и ПК. Отдельное мобильное приложение не требуется.

### Цели системы

- Не терять клиентов из разных источников (ВКонтакте, сайт, телефон)
- Отслеживать производство каждой лодки от первого этапа до передачи покупателю
- Автоматически считать стоимость лодки с учётом всех выплат и штрафов
- Формировать отчёт по затратам и нарушениям в разрезе каждой лодки
- Анализировать клиентов с помощью ИИ и помогать операторам готовить ответы
- Хранить актуальные инструкции и документы компании

### Технические требования

| Параметр | Требование |
|----------|-----------|
| Тип | Веб-приложение (responsive), браузер на ПК и мобильном |
| Мобильная версия | Адаптивный интерфейс, от 320px |
| Авторизация | Email + пароль, JWT-токены, разграничение по ролям |
| Хранение файлов | Облачное хранилище (S3 / Яндекс Облако) |
| База данных | PostgreSQL |
| Безопасность | HTTPS, шифрование паролей, логирование действий |
| Производительность | Загрузка до 2 сек на мобильном 4G |

## 3. Модуль CRM — база клиентов

### 3.1. Источники клиентов

- **ВКонтакте** — автоматический импорт из входящих сообщений через VK Callback API

## 5. Модуль отчётов

### 5.1. Отчёт по лодке

Формируется по номеру лодки. Доступен суперадмину и администратору. Содержит:

- Шапка: номер, модель, цвет, даты, статус
- Этапы с датами, исполнителями, заметками и фото
- Выплаты: этап / сотрудник / сумма
- Бонусы: вид / сотрудник / сумма / статус (выплачен / нет)
- Штрафы: дата / категория / виновный / описание / сумма / фото
- Итоговая таблица: выплаты − штрафы = чистые затраты
- Галерея фото лодки по этапам
- Экспорт в PDF — одна кнопка

### 5.2. Сводный отчёт

- Количество лодок: запущено / завершено / продано
- Суммарные затраты на персонал
- Суммарные штрафы с разбивкой по категориям и сотрудникам
- Средняя стоимость лодки за период
- Экспорт в Excel / CSV

## 7. Интеграции

### 7.1. ВКонтакте — базовая интеграция

- Подключение через VK API (Callback API для входящих сообщений сообщества)
- При новом сообщении — автоматически создаётся карточка клиента
- Дубликаты не создаются — новое сообщение добавляется к существующей карточке

### 7.2. Сохранение переписки ВКонтакте

| Параметр | Описание |
|----------|----------|
| Хранение | Каждое сообщение сохраняется с отметкой времени и направления (входящее / исходящее) |
| Вложения | Фото и файлы из переписки сохраняются в облаке |
| Синхронизация | Импорт истории за последние 90 дней при первом подключении |
| Отображение | Хронологический чат в карточке клиента |
| Поиск | Полнотекстовый поиск по всем сообщениям |
| Хранение | Бессрочно, не удаляется при изменении статуса |

### 7.3. ИИ-модуль подготовки ответов

При поступлении нового сообщения система автоматически готовит оператору предложенный ответ.

| Компонент | Описание |
|-----------|----------|
| Триггер | Новое входящее сообщение в ВК или заявка с сайта с комментарием |
| Контекст для ИИ | Текст сообщения + история переписки + карточка клиента + промпт |
| Нейросеть | Провайдер и модель настраиваются суперадмином. Промпт — отдельно |
| Предложенный ответ | Черновик отображается рядом с сообщением клиента |
| Работа оператора | Принять / отредактировать / написать свой → кнопка «Отправить» |
| Отправка | Через VK API, сохраняется в истории переписки |
| Логирование | Ответ ИИ, финальный ответ, кто отправил, время |

> Промпт для ИИ-модуля настраивается в разделе «Настройки» и поддерживает версионирование.

### 7.4. Краткая сводка о клиенте (ИИ-анализ)

Автоматически формируется на основе переписки и данных карточки. Отображается в верхней части карточки и в виде цветового индикатора в списке клиентов.

| Блок | Содержание |
|------|-----------|
| Температура | 🔴 Горячий / 🟡 Тёплый / 🔵 Холодный — по анализу переписки и активности |
| Что хочет | Резюме запроса клиента: модель, вопросы, пожелания — 1–3 предложения |
| Главное возражение | Основной барьер к покупке (цена, сроки, характеристики, недоверие) |
| Рекомендация | Совет оператору: что ответить, на что сделать акцент |
| Обновление | Автоматически при каждом новом сообщении или вручную по кнопке |

## 12. ИИ-ассистент завода (внутренний чат)

Чат внутри системы где оператор задаёт произвольные вопросы на основе всех данных завода.

### 12.1. Примеры запросов

| Тип | Пример |
|-----|--------|
| Аналитика | «Сколько лодок продано в феврале и на какую сумму выплат?» |
| Клиенты | «Покажи горячих клиентов которые не получили ответ больше 2 дней» |
| Штрафы | «Кто из сборщиков чаще всего получает штрафы за брак?» |
| Поиск по переписке | «Найди клиентов которые спрашивали про рассрочку» |
| Производство | «Сколько мы потратили на сборщиков по лодке №34?» |
| Документы | «Найди договор подряда с Ивановым» |
| Инструкции | «Что делать если шов потёк на этапе сборки бортов?» |
| Сводка | «Какие лодки сейчас в производстве и на каком этапе?» |

### 12.2. Источники данных

- CRM — карточки клиентов, статусы, температура
- Переписки ВК — полные тексты сообщений
- Производство — карточки лодок, этапы, даты
- Штрафы — реестр с категориями и суммами
- Финансы — выплаты, бонусы, итоги
- Документы — текст PDF и DOCX
- Инструкции — тексты всех инструкций

### 12.3. Архитектура — два режима

**Аналитический (SQL через ИИ)** — вопрос → ИИ генерирует SQL → выполняется в БД → ответ. Точные цифры по лодкам, клиентам, штрафам, выплатам.

**Векторный поиск (RAG)** — тексты разбиты на чанки и проиндексированы в pgvector. При запросе находятся похожие фрагменты → передаются в ИИ как контекст → ответ.

> Рекомендуется поэтапная реализация: сначала SQL-режим (2–3 нед.), затем векторный поиск (4–6 нед.)

### 12.4. Интерфейс

- Боковая панель или отдельная страница — доступен из любого раздела
- Ответ содержит ссылки на источники — карточку клиента, лодки или документа
- История диалога сохраняется в сессии
- Логирование всех запросов и ответов

## 13. Офлайн-режим и синхронизация

Гибридная архитектура для работы в условиях нестабильного интернета.

### 13.1. Архитектура

```
Устройства на производстве (телефоны, планшеты, ПК)
                    ↓ Wi-Fi (локальная сеть)
         Локальный сервер на производстве
         (мини-ПК, работает без интернета)
                    ↓ при наличии интернета
              Облачный мастер-сервер
              (VPS, доступен откуда угодно)
```

| Компонент | Описание |
|-----------|----------|
| Локальный сервер | Мини-ПК (Intel NUC или аналог) в локальной сети производства. Работает без интернета. Стоимость: 15 000–25 000 руб. |
| Облачный мастер | VPS в облаке. Полная копия данных. Доступен из офиса, телефона менеджера, любой точки. |
| Синхронизация | Автоматически при появлении интернета. Ручная — по кнопке. |

### 13.2. Разделение по доступности

| Модуль | Доступность | Примечание |
|--------|:-----------:|-----------|
| Этапы производства | Локально | Фото, заметки, отметки |
| Штрафы | Локально | Добавление и просмотр |
| Карточки лодок | Локально | Создание и редактирование |
| Инструкции | Локально | Кешируются при синхронизации |
| CRM — клиенты | Облако | Требует интернет |
| Переписки ВК | Облако | Требует интернет |
| ИИ-ассистент | Облако | Требует интернет |
| Документы | Облако | Требует интернет |
| Отчёты | Оба | Локально по производству, полные — облако |

### 13.3. Логика синхронизации

- Автоматически при обнаружении интернет-соединения
- Интервал фоновой синхронизации — каждые 15 минут
- Конфликты: локальная версия побеждает для производственных данных
- Конфликты: облачная версия побеждает для CRM и документов
- Статус синхронизации отображается в шапке интерфейса
- При ошибке — уведомление суперадмину

### 13.4. Требования к локальному серверу

| Параметр | Требование |
|----------|-----------|
| Железо | Мини-ПК: Intel NUC, Beelink или аналог. RAM 8 ГБ, SSD 256 ГБ |
| ОС | Ubuntu Server 22.04 LTS |
| ПО | Docker — тот же стек что и облачный сервер |
| Сеть | Wi-Fi роутер на производстве |
| Доступ | http://vsz.local или по IP в локальной сети |
| Резервная копия | Ежедневный бэкап на внешний диск + синхронизация с облаком |
