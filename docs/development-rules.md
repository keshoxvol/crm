# Правила разработки

## Миграции базы данных

1. Все изменения схемы БД выполняются только через Liquibase.
2. Каждое изменение оформляется новой миграцией (новый changeset).
3. Уже примененные миграции не изменяются и не удаляются.
4. Миграции должны быть безопасными для данных: нельзя удалять или перезаписывать данные без отдельного согласованного плана.
5. При эволюции схемы используем подход expand/contract:
   - сначала добавляем новые поля/таблицы и обеспечиваем совместимость;
   - переносим чтение/запись на новую схему;
   - только после этого (в отдельной миграции) убираем старые элементы, если это безопасно.

## Тестирование

1. Критический функционал обязательно покрывается тестами.
2. Для бизнес-критичных сценариев минимум:
   - позитивный сценарий;
   - ключевые негативные/пограничные проверки.
3. Исправление багов в критичном функционале сопровождается тестом, который воспроизводит дефект.
